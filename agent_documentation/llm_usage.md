# AI Assistance and Development Notes

## Overview

I used ChatGPT throughout this project as a technical sounding board and engineering assistant. My main goal was to speed up the tedious parts of the development cycle - like writing boilerplate data pipelines, formatting evaluation scripts, and structuring the repository - so I could focus on the actual machine learning decisions.

Everything generated by the AI was manually reviewed, tested, and tweaked before making it into the final codebase. 

### System Details

* **Model:** ChatGPT Plus (using the latest GPT-5.4 model)
* **Provider:** OpenAI
* **Usage Mode:** Interactive development assistance
* **Primary Use Cases:** Debugging, code design, documentation

## How I Used the LLM (Examples & Workflow)

Real-world ML engineering is rarely completely smooth, and I relied on the AI to help navigate several friction points during development.

**The CVAT Export Problem**
The free tier of CVAT does not let you download your annotated images alongside the labels, giving only the bounding box metadata. I asked the model how to automatically attach the annotations back to the original dataset. It helped me draft a Python utility that reads the CVAT XML exports, maps them back to the original PASCAL VOC image directory, and reconstructs the standard YOLO training folder structure, saving me hours of manual file matching.

**Balanced Dataset Sampling**
I needed to isolate a balanced subset of images containing exactly the four classes I cared about, ensuring I had at least 25 objects per class before I started manually annotating. I asked the model to help me design a script for this. We created a script that scans the original VOC annotations, filters for my target classes, and intelligently selects a diverse batch of images until the distribution thresholds were met.

**Architecture Selection**
Initially, I only planned to train the RT-DETR model. During a brainstorming session about the project structure, I asked the model for its thoughts on my approach. It pointed out that comparing a lightweight CNN (YOLOv8n) against a heavier transformer (RT-DETR-l) on a small, constrained dataset would yield a much more interesting engineering analysis. It was right. I pivoted the project to evaluate both, which clearly highlighted the data-hungry nature of transformers.

**GPU Benchmarking Bug**
Getting the PyTorch CUDA timing right for the inference benchmark required digging into asynchronous GPU execution principles. My initial inference benchmark scripts were giving me wildly inconsistent timings. I asked the model to help me debug the issue, and it identified that I was measuring the high-level Python pipeline overhead instead of pure GPU inference time. We rewrote the benchmarking loop to include proper GPU warmup runs, repeated inference passes, and `torch.cuda.synchronize()` calls to ensure stable, accurate latency metrics.

**Colab Portability**
Moving a local dataset up to Google Colab for training is notoriously slow if you upload raw files, and making sure the code ran seamlessly on both a local Windows machine and a Linux-based Colab instance meant being very careful with path handling. I asked the model to help me write a clean bundling script. It generated logic that reads a manifest, zips up only the strictly required images and labels, and outputs a single archive that uploads to Google Drive in seconds.

## What I Would Do With More Time

If I had another week to push this project further, I would focus on several advanced techniques. I actually discussed all of the following concepts with the AI agent during the planning phase, but we ultimately deemed them too advanced or time-consuming for the initial scope of this project:

* **Hyperparameter Optimization:** Implementing automated search methods like grid search or Bayesian optimization instead of relying on default learning rates.
* **Scaling Up Models:** Testing YOLOv8s or RT-DETR-x to see how the parameter count impacts the PR curves on this specific dataset.
* **Advanced Augmentation:** Writing custom data loaders for CutMix, MixUp, or synthetic copy-paste augmentations to force the models to generalize better on occluded objects.
* **Extended Training Schedules:** Giving the RT-DETR model 100+ epochs. Transformers historically need a lot of time to converge, and I kept the schedule short to maintain a fair comparison budget with YOLO.
* **Cross-Validation:** Running K-fold splits to get a more statistically robust performance estimate, rather than relying on a single train/val/test split.
